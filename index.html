项目一：帕姆帮帮（测试版）数据与 AI 合规复盘

基于《崩坏：星穹铁道》公开推出的游戏内 AI 智能对话助手，模拟法务/数据合规团队对已上线功能进行复盘评估、风险识别与优化建议输出。

打开深度研究附件：PIPIA 专项工作底稿

01公开信息与评估结论

公开资料显示，“帕姆帮帮（测试版）”是《崩坏：星穹铁道》4.2 版本期间推出的游戏内 AI 智能对话/助手功能，入口位于游戏聊天列表，功能内容包括星海导航、角色养成方案、泛对话、帕姆换装和每周奖励等模块。

评估结论：已具备若干积极合规设计，但仍应按“中高风险 AI 互动功能”持续治理。尤其要复核内容数据用于模型训练、去标识化标准、未成年人账号策略和 AI 标识留存。
  • 公开说明中已提示对话内容由人工智能技术生成。
  • 测试期结束后暂时下线优化，具备复盘空间。
  • 玩家输入与 AI 回复可能用于内容分析和模型训练。
  • 玩家可在“数据设置”中关闭相关训练用途。
个人信息保护 生成式 AI 训练数据治理 未成年人保护 游戏内 AI 助手
深度研究附件： 本项目另设《PIPIA 专项工作底稿（模拟）》，不重复展示基础功能背景，重点展开处理活动颗粒度、PIPIA 判断链条、风险矩阵、整改验收证据包和复评机制。

03PIPIA 触发判断

PIPIA 的核心功能，是在开展可能对个人权益产生重大影响的个人信息处理活动前，对处理目的、处理方式、个人权益影响及安全风险进行系统评估，并据此形成风险控制措施和留存记录。

敏感个人信息

为什么触发：玩家可能主动输入真实姓名、联系方式、账号密码、健康状况、情感状态、未成年人身份等信息。
复核重点：功能是否提示玩家不要输入敏感信息，是否设置识别、脱敏、拦截和删除机制。

自动化决策

为什么触发：系统可能根据玩家账号状态、角色配置、任务进度自动生成养成建议、导航建议和差异化回复。
复核重点：是否影响玩家资源投入或消费决策，AI 生成属性是否透明，玩家是否可关闭相关个性化或训练用途。

模型训练用途

为什么触发：提供当次回复是一项处理目的，将玩家输入和 AI 回复继续用于内容分析或模型训练，则属于更进一步的数据用途。
复核重点：告知是否充分，关闭路径是否清晰，关闭后是否停止进入训练链路，历史数据如何处理，去标识化是否足够。

第三方与出境

为什么触发：AI 功能可能接入模型供应商、云服务、内容审核、日志分析或客服系统，数据可能离开游戏公司自有控制范围。
复核重点：判断是委托处理、共同处理还是独立处理；若涉及境外模型或境外支持，还需判断数据出境路径。

04AI 合规专项

“帕姆帮帮（测试版）”由 AI 根据玩家输入和游戏场景生成回复，属于生成式 AI 应用场景。公开说明中的 AI 生成提示和数据设置关闭路径是积极设计，但训练数据治理仍是复盘重点。

  • 不得诱导玩家透露真实身份、联系方式、账号密码或住址。
  • 不得以角色身份给出医疗、法律、金融等专业建议。
  • 养成建议不得过度绑定抽卡、充值或限时奖励。
  • 截图、转发或分享到社区时，AI 标识应尽量保留。
训练数据治理重点：
复核“去标识化”是否仍可通过 UID、时间戳、设备、语义内容等重新关联特定玩家；关闭训练用途后，需明确未来数据、已采集数据和已进入训练集数据的处理边界。

05未成年人保护专项控制

默认策略

未成年人账号默认关闭内容数据用于模型训练，不主动推送长时段陪伴式对话。

内容策略

养成建议避免与充值、抽卡、限时奖励形成强绑定，对账号交易、绕过防沉迷等内容严格拒答。

监护人权利

提供查询、删除、投诉、关闭功能的清晰路径，未成年人相关请求进入单独处理 SOP。

审计留痕

儿童个人信息处理应保留更严格的权限审批、访问记录、操作日志和删除记录。留痕不是为了扩大数据保存，而是为了证明“谁在何时、基于什么原因、处理了哪些未成年人相关数据”，便于内部审计、监护人请求处理和监管问询时复盘。

项目二：AIGC 二创工坊 AI 合规评估

本案例为模拟新功能，不代表《崩坏：星穹铁道》已上线该功能。设定基于二次元玩家常见的头像、壁纸、表情包、角色祝福语和米游社配图需求，用于展示生成式 AI 功能上线前的合规评估方法。

01功能设定与评估结论

“AIGC 二创工坊”允许玩家选择《崩坏：星穹铁道》角色、场景或游戏截图，输入简短描述后生成角色主题头像、手机壁纸、表情包、生日贺图、纪念票风格图片或发帖文案。

评估结论：可作为模拟新功能进入灰度测试，但上线前应完成生成式 AI 服务合规评估、生成合成内容标识方案、UGC 内容审核、IP/二创边界控制和未成年人保护策略。
  • 功能核心不再是 AI 对话，而是图文生成和分享传播。
  • 重点风险从“对话数据训练”转为“生成内容标识、二创边界和 UGC 审核”。
  • 默认仅允许个人非商业分享，不开放商用授权或去水印导出。
  • 生成内容不得冒充官方公告、活动海报或客服通知。
AIGC 图片生成 二创生态 生成内容标识 水印方案 UGC 审核

04生成标识与水印方案

水印可以解决“可识别”和“防误认”的一部分问题，但不能替代内容审核、IP 边界和投诉处置。建议采用显式水印、隐式标识和平台标签的组合方案。

开拓者纪念票 · AIGC 二创示例 生成图片右下角固定展示“AI 生成 · 非官方原图 · 仅供个人非商业分享”。
  • 显式水印：导出图片固定添加“AI 生成”“非官方原图”“仅供个人非商业分享”等字样。
  • 隐式标识：在图片元数据中写入生成工具、生成时间和内容 ID，便于平台识别和追溯。
  • 分享保留：分享到米游社或外部平台时，页面标签和图片水印均应保留。
  • 禁止去水印：不提供去水印下载、商用授权导出或“官方物料风格”模板。
  • 举报追踪：每张图绑定生成内容 ID，支持用户举报、撤回分享和违规样本复盘。

项目三：境内第三方服务与 SDK 接入合规评估

本案例模拟《崩坏：星穹铁道》客户端或运营后台接入境内客服系统、境内运营分析/埋点统计服务（自研或第三方）、境内广告 SDK、崩溃日志 SDK 和内容安全/风控服务，输出第三方处理关系、SDK 字段台账、最小必要审查和境外访问排除检查。

01场景设定与初步结论

本项目聚焦境内法务方向：第三方服务与 SDK 接入本身并不等于数据出境，核心是先完成境内第三方接入合规审查，再完成数据出境排除项检查。

评估结论：可按境内第三方服务与 SDK 接入流程进行评估。重点不是安全评估/标准合同路径，而是第三方处理关系、采集字段最小必要、SDK 超范围采集检测、用户告知、委托处理协议、二次共享限制和权限/日志留存控制。
  • 境内第三方 SDK 仍需审查收集字段、调用权限、处理目的和共享范围。
  • 同一 SDK 可能同时存在委托处理、独立处理或共同处理风险，需要逐项判断。
  • 若未发现境外服务器接收、境外远程访问或境外关联方处理，可记录为“不进入数据出境路径”。
  • 上线前应形成 SDK 清单、字段台账、技术检测记录、合同条款和隐私政策更新记录。
境内第三方 SDK 尽调 最小必要 委托处理 供应商尽调

03境外访问排除检查

本项目主线是境内第三方接入，但仍需保留出境排查记录。排除检查的目标是证明：本次 SDK 接入未发现境外服务器接收、境外远程访问、境外客服处理或境外关联方处理个人信息。

Step 1

确认主体与服务器位置。供应商是否为境内主体，数据接收服务器、备份服务器、日志服务器和客服系统是否均位于中国境内。

Step 2

确认访问权限。是否存在境外工程师、境外客服、境外母公司/关联公司或境外供应商通过后台、工单、日志平台远程访问境内玩家个人信息。

Step 3

确认二次共享路径。供应商是否将数据同步给境外子处理者、境外云服务、境外广告平台、境外分析工具或境外技术支持团队。

Step 4

形成排除结论。若上述检查均为否，可记录为“本项目按境内第三方服务/SDK 接入管理，暂不进入数据出境安全评估、标准合同或认证路径”；若任一项为是，则启动数据出境专项评估。

04第三方处理关系判断表

委托处理

第三方仅按游戏公司指令处理数据，不自行决定处理目的和方式。重点签署委托处理协议，约定保密、安全、删除、审计、事件通知和禁止转委托。

共同处理

游戏公司与第三方共同决定处理目的和方式。重点明确各方责任、用户权利响应分工、告知内容和发生争议时的责任边界。

独立处理

第三方独立决定数据用途，例如自建画像、广告投放或跨平台归因。重点审查是否需要单独告知/同意、是否构成第三方提供和是否允许接入。

      04AI Agent 扩展方案

      若后续接入 AI Agent，建议采用“规则引擎先硬触发 + AI Agent 再复核追问”的双层结构。注意：API 密钥不能写进静态网页前端，应通过 Vercel/Netlify serverless function 或后端代理调用。

      Agent 1：事实补全

      检查功能描述是否缺少字段、触发时机、第三方、留存期限、用户告知和退出路径，并生成追问清单。

      Agent 2：法律风险复核

      基于规则命中结果，从 PIPIA、AI 标识、未成年人、SDK、出境和自动化决策角度交叉验证风险。

      Agent 3：整改建议生成

      把风险点转成可执行整改项，例如字段删除、弹窗提示、合同条款、日志脱敏、留存期限和验收标准。

      # Python 规则引擎示意 def assess_feature(text, flags): risks = [] if any(word in text for word in ["实名", "面部识别", "联系方式", "订单", "未成年人"]): risks.append("可能涉及敏感或特殊保护个人信息，建议进行 PIPIA 初筛") if flags.get("ai") or any(word in text for word in ["AI", "生成图片", "生成文案", "Prompt"]): risks.append("涉及生成式 AI 功能，需评估内容安全、标识和投诉机制") if flags.get("training") or any(word in text for word in ["训练", "模型优化", "样本回流"]): risks.append("涉及训练数据治理，需明确告知、去标识化和退出机制") if flags.get("third_party") or any(word in text for word in ["SDK", "第三方", "客服系统"]): risks.append("涉及第三方处理，应判断委托处理、共同处理或独立处理") return risks
